# Cloudflare Deployment

`comical-tool` uses two Workers in Cloudflare:

- `comical-tool-api` from `worker/`, serving `tool.sqlboy.me/api/*` and `tool.sqlboy.me/short/*`.
- `comical-tool-web` from `web/`, serving `tool.sqlboy.me/*` through OpenNext.

## Resources

The API Worker expects these bindings:

- D1 database `comical-tool` with migrations from `migrations/d1`.
- R2 bucket `comical-tool-objects` for images and temporary files.
- KV namespace `KV` for clipboard data and other volatile state.

The Web Worker expects:

- static assets binding generated by OpenNext.
- `NEXT_PUBLIC_API_BASE_URL=https://tool.sqlboy.me`.

## Manual Deploy

Deploy the API Worker first so API routes are in place before the Web route catches
the zone wildcard.

```bash
cd worker
npm run deploy
```

Deploy the Web Worker:

```bash
cd web
npm run deploy
```

Smoke check:

```bash
curl -I https://tool.sqlboy.me
curl -sS -D - https://tool.sqlboy.me/api/health
curl -sS -D - https://tool.sqlboy.me/short/__missing_smoke_check__
```

Expected results:

- `/` returns an HTML response with `x-opennext: 1`.
- `/api/*` returns JSON from `comical-tool-api`.
- `/short/*` returns redirect or JSON error from `comical-tool-api`.

## Runtime Differences

Cloudflare runtime intentionally differs from the self-hosted Go server in these
areas:

- Clipboard records live in KV instead of the relational database.
- Uploaded object bytes live in R2 instead of local storage.
- D1 stores API metadata using integer epoch timestamps.
- Cleanup runs from Worker cron instead of a self-hosted background job.
- Hertz is not run inside Workers; `worker/` is a Cloudflare-specific adapter.

Keep API paths and JSON shapes aligned with the self-hosted server. When behavior
must differ because of Cloudflare bindings, document it in this file.
